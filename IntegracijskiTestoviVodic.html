Pitanje 1

Što su integracijski testovi API-ja?

Integracijski test provjerava više dijelova aplikacije zajedno, a ne jednu izoliranu metodu.

Definicija

Integracijski test API-ja pokreće aplikaciju skoro kao pravi server i šalje HTTP zahtjev preko HttpClient-a. Time se zajedno testiraju ruta, controller, model binding, validacija, autorizacija, servisni sloj ako postoji i pristup bazi.

Razlika od unit testa

Unit test bi direktno instancirao controller i pozvao metodu, npr. controller.Get(1). To ovdje ne radimo. Naš test zove /api/guitars/1, pa ASP.NET Core sam prolazi kroz routing, filtere i middleware.

Gotov odgovor za profesora

Integracijski testovi u našem projektu testiraju stvarne API endpointove preko HttpClienta. Ne instanciramo controller ručno i ne mockamo DbContext, nego pokrećemo aplikaciju u Testing environmentu pomoću WebApplicationFactory. Stvarnu development bazu zamjenjujemo EF Core InMemory bazom kako bi testovi bili izolirani, ponovljivi i sigurni.

Pitanje 4

Što radi CustomWebApplicationFactory?

To je srce integracijskog testiranja u ASP.NET Core aplikaciji.

Uloga klase

CustomWebApplicationFactory : WebApplicationFactory<Program> pokreće naš web projekt za potrebe testova. Ona kaže aplikaciji da radi u Testing environmentu i mijenja produkcijske servise testnim servisima.

Zašto je to važno?

Bez ove klase test bi mogao slučajno koristiti stvarnu development bazu, stvarni login ili stvarne vanjske servise. Mi to izbjegavamo tako da u testovima zamijenimo AppDbContext i autentikaciju.

builder.UseEnvironment("Testing");

builder.ConfigureTestServices(services =>
{
    services.RemoveAll<AppDbContext>();
    services.RemoveAll<DbContextOptions<AppDbContext>>();

    services.AddDbContext<AppDbContext>(options =>
        options.UseInMemoryDatabase(_databaseName));
});

Gotov odgovor za papir

CustomWebApplicationFactory nam omogućuje da testovi pokrenu stvarnu ASP.NET Core aplikaciju, ali s testnom konfiguracijom. U njoj se postavlja Testing environment, stvarni AppDbContext se uklanja iz dependency injection containera, a umjesto njega se dodaje EF Core InMemory baza. Tako testovi ne diraju razvojnu bazu.
Pitanje 5

Kako rade testna baza i TestDataSeeder?

Svaki test dobiva poznato početno stanje baze.

InMemory baza

Koristimo UseInMemoryDatabase jer želimo brze i izolirane testove. To nije stvarna PostgreSQL baza, nego EF Core provider koji čuva podatke u memoriji tijekom testa.

Seed podaci

TestDataSeeder.SeedApiDataAsync prvo briše i ponovno kreira testnu bazu, zatim dodaje poslovnicu, korisnika, stranku, zaposlenika, tehničara, gitaru, znanje i servisni nalog. Zbog toga svaki test zna koji ID postoji.

await db.Database.EnsureDeletedAsync();
await db.Database.EnsureCreatedAsync();

db.Poslovnice.Add(new Poslovnica
{
    Id = ExistingOfficeId,
    Ime = "Test poslovnica",
    Adresa = "Test adresa 1"
});

await db.SaveChangesAsync();

Zašto imamo konstante?

Konstante poput ExistingGuitarId, ExistingCustomerId i MissingLongId čine testove čitljivima. Umjesto magičnog broja 999, test jasno kaže da traži nepostojeći ID.

Pitanje 6

Kako testiramo autorizaciju bez pravog login-a?

U integracijskim testovima ne želimo otvarati browser ni prolaziti Google login.

TestAuthHandler

TestAuthHandler čita header X-Test-Role. Ako header postoji, test napravi korisnika s tom rolom. Ako header ne postoji, korisnik je neprijavljen.

ApiTestBase helper

CreateAuthenticatedClient() vraća HttpClient s ulogom Manager po defaultu. Ako treba Admin, zovemo CreateAuthenticatedClient(IdentitySeed.AdminRole).

protected HttpClient CreateAuthenticatedClient(string role = IdentitySeed.ManagerRole)
{
    var client = Factory.CreateClient();
    client.DefaultRequestHeaders.Add(TestAuthHandler.RoleHeaderName, role);
    return client;
}

Što time provjeravamo?

Ako endpoint ima [Authorize], neprijavljeni klijent dobiva 401 Unauthorized. Ako endpoint traži Admin rolu, a klijent ima Manager rolu, dobiva 403 Forbidden. Time testiramo stvarna autorizacijska pravila bez stvarnog cookieja ili Google OAuth procesa.

Pitanje 8

Primjer iz našeg projekta: Guitars API

Ovo je dobar obrazac za objašnjenje profesoru jer pokriva uspjeh, 404 i 400.

Uspješno kreiranje

Test šalje POST na /api/guitars s validnim GuitarRequestDto. Očekuje 201 Created, čita vraćeni GuitarDto i zatim provjerava da je zapis stvarno spremljen u testnoj bazi.

Validacijska greška

Za BadRequest namjerno stavimo nepostojeći MarkaId ili KupacId. API mora odbiti zahtjev jer povezani zapis ne postoji.

[Fact]
public async Task Create_WhenRequestIsValid_ReturnsCreatedAndPersistsGuitar()
{
    await TestDataSeeder.SeedApiDataAsync(Factory);
    var request = ValidRequest("NEW-GUITAR-001");

    var response = await CreateAuthenticatedClient()
        .PostAsJsonAsync("/api/guitars", request);

    response.StatusCode.Should().Be(HttpStatusCode.Created);
    var created = await response.Content.ReadFromJsonAsync<GuitarDto>();
    created!.SerijskiBroj.Should().Be(request.SerijskiBroj);
    (await FindAsync<Gitara>(created.Id))!.SerijskiBroj
        .Should().Be(request.SerijskiBroj);
}

Kako to objasniti

Ovaj test ne provjerava samo da controller vrati Created. On provjerava cijeli tok: HTTP POST, JSON model binding, validaciju DTO-a, rad controllera, EF Core spremanje i response DTO. Zato je to integracijski test.

Pitanje 10

Kako pokrenuti testove i kako ih objasniti?

Najkraći dokaz da sve radi je dotnet test.

Naredba

Iz root foldera repozitorija pokreni:

dotnet test

Ako želiš ciljano samo test projekt:

dotnet test "Servis Centar Za Gitare.Tests\Servis Centar Za Gitare.Tests.csproj"

Kratko objašnjenje za usmeni

Testovi pokreću aplikaciju pomoću WebApplicationFactory, mijenjaju stvarnu bazu s InMemory bazom i koriste TestAuthHandler za simulaciju prijavljenog korisnika s rolom. Svaki test puni bazu poznatim podacima, šalje stvarni HTTP zahtjev preko HttpClienta i provjerava status code, response body i ponekad zapis u bazi.

Česte greške koje treba izbjeći

  • Direktno pozivanje controllera umjesto HTTP endpointa.
  • Korištenje stvarne development baze u testovima.
  • Test bez seedanja podataka, pa rezultat ovisi o starom stanju baze.
  • Testiranje samo uspješnih scenarija bez 404 i 400.
  • Zaboraviti autorizaciju, posebno razliku između 401 i 403.
